# 项目规范

---

## 一、项目包结构

基础包根据项目自定义，下分 `common` 和 `module` 两个顶层包。

### 1.1 common — 通用基础包
业务无关，可直接移植到其他项目复用。包括：
- `config/` — 通用配置类（如跨域、序列化配置）
- `constants/` — 通用常量（如通用状态码、分页默认值）
- `enums/` — 通用枚举（如启用/禁用状态、是/否等）
- `exception/` — 通用异常定义与全局异常处理
- `response/` — 统一返回体封装
- `util/` — 纯技术性工具类

### 1.2 module — 业务模块包
项目特有的业务逻辑。以下目录结构为**基础规范**，按需取用：

- `constants/` — 业务常量（如订单状态、业务特定阈值）
- `enums/` — 业务枚举（如订单类型、用户角色等）
- `controller/` — REST 控制器
- `business/` — Service 编排层
- `service/` — 领域逻辑（接口 + impl 实现类）
- `mapper/` — 数据访问
- `pojo/` — BO、DTO、Entity、VO、Request、Converter
- `util/` — 业务工具类（静态方法，不依赖 Spring Bean，如从 UA 解析设备编号、解析告警描述）
- `filter/` — 过滤器
- `interceptor/` — 拦截器
- `aop/` — 切面逻辑（如日志、权限、限流等）
- `init/` — 初始化逻辑
- `listener/` — 事件监听器
- `task/` — 定时任务

#### 产品形态扩展
当项目存在多种产品形态（如标准版、企业版、SaaS 版）时，`module` 下可按产品形态划分子包，每种产品形态各自包含完整的一套目录结构。例如：

```
module/
├── standard/          # 标准版
│   ├── controller/
│   ├── business/
│   ├── service/
│   ├── mapper/
│   └── pojo/
└── enterprise/        # 企业版
    ├── controller/
    ├── business/
    ├── service/
    ├── mapper/
    └── pojo/
```

原则：
- **不抽取产品间公共模块**，选择冗余，保持各产品形态完全独立
- 产品形态下的类命名需体现归属，如 `StandardUserController`、`EnterpriseUserController`
- `module` 下的目录以基础规范为参照，允许根据项目实际情况扩展或调整，但不违背 `common` 与 `module` 的分层边界

### 1.3 common 与 module 的关系
`constants`、`enums`、`exception` 等在两个包中同时存在是正常的 — `common` 中放的是通用定义，`module` 中放的是业务特定定义。判断标准：**该内容脱离当前项目是否还有意义**，有意义放 `common`，无意义放 `module`。

## 二、POJO 分类与数据流转

### 2.1 POJO 分类定义

#### Request
- 前端请求体的接收对象
- 命名规则：`xxxRequest`
- 可包含参数校验注解（如 `@NotNull`、`@Size`）
- 注意：Controller 的入参不限于 Request 对象，`@RequestHeader`、`@PathVariable`、`@RequestParam` 等基本类型参数可直接获取，只有请求体对象必须归一为 Request

#### VO（View Object）
- Controller 的视图返回对象
- 命名规则：`xxxVO`
- 用于封装返回给前端的数据结构

#### DTO（Data Transfer Object）
- 仅作为 Business 层的入参对象存在
- 命名规则：`xxxDTO`
- 当 Business 方法需要两个及以上参数时，封装为 DTO 传入；单个参数可直接用基本类型

#### BO（Business Object）
- 业务对象，贯穿整个业务处理流程
- 命名规则：`xxxBO`
- 项目中所有业务私有方法、Service 的公共/私有方法、中间处理对象，涉及的参数和返回值都应使用 BO
- BO 是业务层的核心数据载体
- **创建后无需修改字段的 BO 优先使用 JDK record**（不可变、零依赖、线程安全），需要中途修改字段的用 Lombok `@Data`

#### Entity
- 数据库实体对象，与数据库表结构一一对应
- 命名规则：`xxxEntity` 或 `xxx`（视项目约定）
- 仅在 Mapper 层使用
- 每个字段必须有中文注释，说明该字段的业务含义
- **必须使用 Lombok `@Data`**（MyBatis-Plus 需要无参构造和 setter）

#### 简化规则
当参数为单个基本类型（如 `Long id`、`String name`）时，无需为其创建专门的 POJO 类，直接传递基本类型即可。两个及以上参数时必须封装为对应的 POJO 对象。

### 2.2 数据流转示意

```
Request → (DTO) → BO → Entity
                                ↓
                              Database
                                ↓
Entity → BO → VO
```

正向：Controller 接收 Request，按需转换为 DTO（或直接传基本类型），Business 转换为 BO 调用 Service，Service 转换为 Entity 持久化。

反向：Mapper 返回 Entity，Service 转换为 BO，Business 返回 BO（或基本类型），Controller 通过 MapStruct 转换为 VO 返回前端。

## 三、分层调用规范

```
Controller / Task / Listener → Business → Service → Mapper
```

Controller、Task（定时任务）、Listener（事件监听）均为应用入口，必须通过 Business 层进入业务逻辑，禁止直接调用 Service 或 Mapper。

### 3.1 入口层

#### Controller
- 入参：请求体必须是 `xxxRequest`，其他参数可通过注解直接获取（基本类型）
- 返回：必须是 `xxxVO`
- 规则：**最严格，POJO 类型不可替换**

#### Task / Listener
- 入参出参不做严格 POJO 约束，根据实际场景灵活定义
- 规则：**必须调用 Business 层，禁止直接调用 Service 或 Mapper**

### 3.2 Business
- 入参：两个及以上参数用 DTO 封装，单个参数用基本类型
- 返回：BO 或基本类型
- 职责：跨 Service 编排，隔离 Service 间的循环依赖

### 3.3 Service
- 入参：两个及以上参数用 BO 封装，单个参数用基本类型
- 返回：BO 或基本类型
- 职责：单一领域逻辑处理
- **必须拆分为接口 + 实现类**：
  - 接口（`service/XxxService.java`）：定义业务方法签名，入参出参遵守 BO 或基本类型规范
  - 实现类（`service/impl/XxxServiceImpl.java`）：实现接口，处理具体逻辑
- 分为两类：
  - **数据库相关 Service**：实现类继承 MyBatis-Plus 的 `ServiceImpl<Mapper, Entity>`，内部通过 Entity 与 Mapper 交互，对外暴露 BO
  - **数据库无关 Service**：纯业务逻辑，不操作数据库，不继承 `ServiceImpl`，入参出参同样遵守 BO 或基本类型规范

### 3.4 Mapper
- 入参：Entity 或基本类型
- 返回：Entity 或基本类型
- 规则：**严格，只允许 Entity 和基本类型**

### 3.5 方法放置规则

业务逻辑写在哪里，取决于其使用范围和性质：

| 场景 | 放置位置 | 说明 |
|---|---|---|
| 仅在当前 Business 内部使用 | Business 私有方法 | 入参出参使用 BO 或基本类型 |
| 不包含业务逻辑，纯技术性质 | 工具类（`common/util/`） | 如字符串处理、日期格式化等，无 Spring 依赖 |
| 包含业务逻辑，但不依赖 Spring Bean | 业务工具类（`module/util/`） | 静态方法，如从 UA 解析设备编号、解析告警描述文件 |
| 可复用且与业务相关，需要 Spring Bean | 定义为 Service | 包含数据库操作的继承 `ServiceImpl`，否则定义独立的 Service |

## 四、MyBatis-Plus 规范

- `@MapperScan` 注解加在启动类上，统一扫描 Mapper 接口，**禁止在 Mapper 接口上使用 `@Mapper` 注解**
- Entity 实体类每个字段必须有中文注释，说明该字段的业务含义
- `ServiceImpl` 自带大量现成方法（如 `save()`、`getById()`、`list()`、`updateById()` 等），入参出参均为 Entity。**禁止在 Service 层之外直接调用这些方法**

正确做法：在自定义的 Service 中新建方法，入参出参遵守 BO 或基本类型规范，内部委托调用 `ServiceImpl` 的已有方法。例如：

```java
// 正确：Service 方法对外暴露 BO，内部委托 ServiceImpl
public UserBO getUserById(Long id) {
    UserEntity entity = this.getById(id);
    return userConverter.toBO(entity);
}

// 错误：Business 层直接调用 ServiceImpl 的 Entity 方法
userServiceImpl.getById(id);
```

## 五、响应与异常规范

### 5.1 统一返回体
包路径：`common.response`（不是 `common.util.response`）

基于 `BaseResponse` 抽象类，通过枚举管理状态，提供三种具体响应类型：

```
BaseResponse（基类，status 枚举 + message）
├── SingleResponse<T>    单条数据，字段：T record
├── MultiResponse<T>     集合数据，字段：List<T> records
└── PageResponse<T>      分页数据，字段：PageInfo<T> data
```

**BaseResponse**：
- status 使用枚举（非 Integer），至少包含 SUCCESS / FAIL / ERROR
- 子类提供统一的静态工厂方法：`ofSuccess()` / `ofFail()` / `ofError()`，命名保持一致

**PageResponse.PageInfo**：
- 字段：current（当前页）、size（每页大小）、pages（总页数）、total（总记录数）、records（数据列表）
- 作为 PageResponse 内部类或独立类，根据复用需要决定

**ofFail 设计**：
- 支持传入错误码 + 消息，前端根据错误码做差异化处理
- `ofFail(int code, String message)` 或 `ofFail(ResponseEnum responseEnum)`

**Controller 返回示例**：
```java
@GetMapping("/{id}")
public SingleResponse<UserVO> getUser(@PathVariable Long id) {
    UserBO bo = userBusiness.getUserById(id);
    return SingleResponse.ofSuccess(userConverter.toVO(bo), "查询成功");
}
```

### 5.2 异常体系
异常必须按类型细化，不同场景抛出对应的异常类型，由全局异常处理器（`@RestControllerAdvice`）统一拦截并返回标准响应。异常分类：

| 异常类型 | 场景 | 示例 |
|---|---|---|
| 校验异常 | 参数校验不通过、业务规则校验失败 | `ValidationException` |
| 中间件异常 | Redis、MQ、数据库等中间件不可用或超时 | `MiddlewareException` |
| 系统异常 | 未预期的内部错误、空指针等 | `SystemException` |

业务异常（`module/exception/`）继承通用异常基类，按业务需要扩展。全局异常处理器按异常类型优先级从具体到通用的顺序 catch，确保每种异常都有对应的处理逻辑和响应码。

## 六、工程约束

### 6.1 收敛原则
以下组件各只允许存在一个实例，逻辑在单个类内部组织，禁止拆分为多个同类组件：
- **Filter** — 全局只一个 Filter，通过内部条件判断区分不同处理逻辑
- **Interceptor** — 全局只一个 Interceptor，通过内部条件判断区分不同拦截逻辑
- **Init** — 全局只一个初始化类，按顺序统一完成加载，避免 Bean 加载顺序冲突

### 6.2 配置文件规范
项目自定义配置必须收敛在一个统一前缀下，通过二级 key 分类，禁止分散配置。例如：

```yaml
pojo-project:
  filter:
    exclude-paths: /login,/register
  cors:
    allowed-origins: "*"
  thread-pool:
    core-size: 4
```

自定义配置的 Java 映射使用 `@ConfigurationProperties(prefix = "xxx.yyy")` 绑定，配置类放置规则：
- **模块专属配置** → `module/{product}/config/`（如 `module/im/config/IMPathProperties.java`）
- **多模块共享配置** → `common/config/`

配置类不属于 POJO（Entity/BO/DTO/VO/Request），不放在 `pojo` 包下。通过 `@EnableConfigurationProperties` 在启动类或配置类上激活。

### 6.3 技术选型约束
- **POJO 转换**：使用 MapStruct，禁止手动 BeanUtil.copyProperties 或逐字段 set
- **工具类**：优先使用 Hutool 提供的工具方法，避免自行实现通用逻辑，便于统一管理和升级

### 6.4 注释与日志规范
- 每个类、接口必须有中文类注释，说明其职责或用途
- 代码注释必须详细，关键逻辑行之间应有中文注释描述意图
- 日志输出使用中文，包括 info、warn、error 级别的日志消息
- Entity 实体类每个字段必须有中文注释（见第四章）
- record 类的字段注释通过类级别 Javadoc 的 `@param` 标签书写：
```java
/**
 * Kafka topic 元信息
 *
 * @param topic Kafka topic 名称
 * @param partition 分区数
 * @param replication 副本数
 */
public record TopicInfoBO(String topic, int partition, int replication) {
}
```

## 七、单元测试规范

### 7.1 核心原则
- **禁止使用 Mock**，所有测试基于 Spring 上下文，使用真实对象
- 覆盖率目标：**100%**，但以下情况可豁免，不计入覆盖率统计：
  - BO / VO / DTO / Request 的 getter / setter / 构造方法
  - MapStruct 生成的转换代码
  - 启动类（Application）
- 豁免方式：通过 JaCoCo 配置 exclude 过滤对应类或方法
- 使用 JaCoCo 插件进行覆盖率检测
- 测试与启动使用同一套配置，不做环境隔离

### 7.2 测试类规范

#### 命名
- 测试类：`被测类名Test`，如 `UserServiceTest`、`OrderBusinessTest`
- 测试方法：`test_方法名_场景_预期结果`，如 `test_getUserById_用户不存在_抛出异常`
- 测试方法必须有中文注释说明测试意图

#### 目录结构
测试类目录与 `src/main/java` 完全镜像：

```
src/test/java/
└── com.xxx/
    ├── common/
    │   └── util/
    │       └── StringUtilTest.java
    └── module/
        ├── controller/
        │   └── UserControllerTest.java
        ├── business/
        │   └── UserBusinessTest.java
        ├── service/
        │   └── UserServiceTest.java
        └── mapper/
            └── UserMapperTest.java
```

### 7.3 各层测试方式

所有测试均使用 `@SpringBootTest` 加载完整上下文，注入真实 Bean：

| 层 | 测试方式 | 说明 |
|---|---|---|
| Controller | `@SpringBootTest` + `MockMvc`（真实 Bean，非 Mock 对象） | 验证 HTTP 入参出参、参数校验、路由 |
| Business | `@SpringBootTest` + 直接注入 | 验证 Service 编排逻辑 |
| Service | `@SpringBootTest` + 直接注入 | 验证业务逻辑、数据转换 |
| Mapper | `@SpringBootTest` + 直接注入 | 验证 SQL、CRUD 操作，使用内存数据库 |
| Util | `@SpringBootTest` 或普通 JUnit | 纯工具方法可脱离上下文 |

### 7.4 测试示例

```java
@SpringBootTest
class UserServiceTest {

    @Autowired
    private UserService userService;

    // 测试：根据ID查询用户 - 正常返回
    @Test
    void test_getUserById_正常返回() {
        // 准备测试数据
        Long userId = 1L;

        // 执行
        UserBO result = userService.getUserById(userId);

        // 验证
        assertNotNull(result);
        assertEquals(userId, result.getId());
    }
}
```
